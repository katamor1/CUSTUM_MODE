はい。新規の小規模機能追加では、AIエージェントには **「調査 → 実装計画 → 限定実装 → 自己レビュー」** の順で依頼するのがよいです。

大事なのは、最初から「この機能を追加して」と丸投げしないことです。
このコードベースだと、共有ヘッダ・グローバル状態・モジュール境界を見落とすと危ないので、**AIにはまず“変更してよい場所を探す仕事”をさせる**のが安全です。

---

## 基本方針

小規模機能追加では、AIエージェントの役割をこう分けるのがよいです。

| 段階        | AIにさせること                  | コード変更 |
| --------- | ------------------------- | ----- |
| 1. 調査     | 関連ファイル・関数・構造体・グローバル変数を特定  | なし    |
| 2. 設計     | 変更方針、変更ファイル、影響範囲、テスト観点を出す | なし    |
| 3. 実装     | 承認した方針だけを実装               | あり    |
| 4. 自己レビュー | 差分、リスク、ビルド・テスト結果を確認       | 原則なし  |

ポイントは、**AIに“最初から編集権限を渡さない”**ことです。
最初の1回は必ず読み取り専用で「どこを変えるべきか」を出させます。

---

## 小規模機能追加の依頼単位

この規模のCコードでは、「小規模」は機能の見た目ではなく、**変更範囲**で定義した方がよいです。

目安はこうです。

| 区分            | 小規模として扱いやすい              |
| ------------- | ------------------------ |
| 変更ファイル数       | 1〜3ファイル                  |
| rawで読ませるファイル数 | 2〜6ファイル                  |
| 共有ヘッダ変更       | 原則避ける。必要なら別タスク化          |
| グローバル変数追加     | 原則禁止。必要なら人間承認            |
| 既存構造体変更       | 原則注意扱い。ABI・保存領域・初期化影響を確認 |
| テスト           | 既存の近傍テスト、または手順ベースの確認でよい  |

つまり、AIに頼むときは、

> 「この機能を追加して」
> ではなく、
> 「この機能を、既存のこのモジュール内に閉じて追加できるか調査して。共有ヘッダ変更やグローバル追加が必要なら実装せず報告して」

という形が安全です。

## 依頼テンプレート 1：最初の調査依頼

最初はこれを使うのがよいです。
この段階では **コード変更禁止** にします。

AIエージェントへの依頼：

以下の小規模機能を追加したい。まずはコードを変更せず、関連箇所の調査だけを行ってください。

## 追加したい機能

* 機能名：
* 目的：
* 期待する動作：
* 既存動作から変えたい点：
* 変えてはいけない点：

## 前提

* 対象はVC6時代のC言語コードです。
* 共有ヘッダ、構造体、グローバル変数への影響を特に注意してください。
* 既存のコーディングスタイル、命名、エラー処理、ログ出力方式に合わせてください。
* まだコード変更はしないでください。

## 調査してほしいこと

1. 変更候補となるモジュール、ファイル、関数を列挙してください。
2. 関連する構造体、共有ヘッダ、マクロ、定数を列挙してください。
3. 読み書きされるグローバル変数があれば、所有元・初期化箇所・主な更新箇所を整理してください。
4. 既存処理の流れを簡潔に説明してください。
5. 変更案を2案まで提示してください。
6. 共有ヘッダ変更、構造体変更、グローバル変数追加が必要な場合は、実装せずに「要承認」として報告してください。
7. 推奨する実装方針、変更ファイル、テスト観点を出してください。

## 出力形式

* 関連ファイル一覧
* 関連関数一覧
* 関連グローバル変数一覧
* 既存処理フロー
* 実装方針案
* リスク
* テスト観点
* 次に実装してよい最小差分

この依頼で、AIにいきなり暴走させず、まず「地図」を作らせます。
レガシーCではここが一番大事です。コードを書く前に沼の深さを測る、みたいなやつです。

## 依頼テンプレート 2：実装依頼

調査結果を見て、人間が方針を選んだ後に実装させます。

AIエージェントへの依頼：

前回の調査結果に基づき、以下の方針で小規模実装を行ってください。

## 採用する方針

* 方針：
* 変更対象ファイル：
* 変更対象関数：
* 変更してよい範囲：

## 制約

* 共有ヘッダは変更しないでください。
* 既存構造体のレイアウトは変更しないでください。
* 新しいグローバル変数は追加しないでください。
* 既存の初期化順序、エラー処理、ログ出力方式を崩さないでください。
* 関係ないリファクタリング、整形、命名変更はしないでください。
* VC6でコンパイル可能なCとして実装してください。
* C99以降の機能は使わないでください。
* 変更は最小差分にしてください。

## 実装手順

1. 変更前に、変更対象関数の現在の責務を短く再確認してください。
2. 最小差分で実装してください。
3. 変更後、差分の意図を説明してください。
4. ビルドエラーが出る可能性がある箇所を自己点検してください。
5. テスト観点を提示してください。

## 出力形式

* 変更ファイル
* 変更関数
* 変更内容の要約
* 変更しなかった候補と理由
* 想定リスク
* ビルド確認結果
* テスト観点

このテンプレートでは、**「共有ヘッダ変更なし」「構造体変更なし」「グローバル追加なし」**を明示しています。
小規模機能追加では、この3つを禁止しておくだけで事故率がかなり下がります。

## 依頼テンプレート 3：レビュー依頼

実装後は、同じAIでもよいですが、できれば別セッション・別モードでレビューさせるとよいです。

AIエージェントへの依頼：

以下の差分をレビューしてください。実装の追加や修正はまだ行わず、指摘と修正案の提示だけをしてください。

## レビュー観点

1. 仕様どおりの動作になっているか
2. 既存動作を壊していないか
3. 共有ヘッダ、構造体、グローバル変数への影響がないか
4. 初期化順序、状態遷移、エラー処理に問題がないか
5. VC6/C言語として問題がないか
6. 配列境界、NULL、未初期化、符号付き/符号なし、文字列終端に問題がないか
7. 関係ないリファクタリングや整形が混入していないか
8. テスト観点に漏れがないか

## 出力形式

* 総評
* 致命的リスク
* 要修正
* 確認推奨
* 問題なしと判断した点
* 追加すべきテスト
* 修正が必要な場合の最小修正案

レビュー時は、AIに「修正までやって」と言わず、まず指摘だけにします。
指摘を見てから、修正を別依頼にした方が差分が膨らみにくいです。

---

## AIへの依頼で必ず入れるべき情報

新規機能追加では、最低限これを渡すと精度が上がります。

| 項目        | 例                        |
| --------- | ------------------------ |
| 目的        | 何のための機能か                 |
| 入力条件      | どの画面、コマンド、信号、ファイル、状態で動くか |
| 出力・副作用    | 何を表示、保存、送信、更新するか         |
| 既存類似機能    | 「○○と似た処理」「△△関数と同じログ方式」   |
| 変更禁止範囲    | 共有ヘッダ禁止、構造体変更禁止など        |
| 対象モジュール候補 | 28モジュール中どこに閉じたいか         |
| 期待する差分サイズ | 1〜3ファイル、関数追加1個程度など       |
| ビルド条件     | VC6、C90相当、既存マクロ使用        |
| テスト方法     | 操作手順、ログ確認、既存テスト名など       |

特に重要なのは **「既存類似機能」** です。
AIは既存の似た処理を見つけられると、コーディングスタイル・エラー処理・ログ出力を合わせやすくなります。

---

## 逆に避けたい依頼

これは避けた方がよいです。

> ○○機能を追加してください。関連ファイルを調べて実装してください。

この書き方だと、AIが広く探索しすぎたり、勝手に共通ヘッダを変えたり、不要なリファクタリングを混ぜる可能性があります。

より良い書き方はこうです。

> ○○機能を追加したいです。まずコード変更せず、既存の類似処理と変更候補を調査してください。共有ヘッダ変更・構造体変更・グローバル追加が必要な場合は実装せず報告してください。変更は対象モジュール内に閉じる前提で、最小差分案を提示してください。

この差はけっこう大きいです。
AIに自由を与える場所は「調査」と「案出し」で、**編集権限は狭く渡す**のがコツです。

## 実運用でのおすすめフロー

日常の小規模追加なら、この流れがよいです。

1. 人間が「薄い仕様」を書く
2. AIに読み取り専用で調査させる
3. AIに変更案を2案まで出させる
4. 人間が1案を選ぶ
5. AIに最小差分で実装させる
6. ビルドログをAIに渡して修正させる
7. 別依頼としてレビューさせる
8. 人間が最終確認する

ここでの重要ルールは、**1回の依頼で“調査・実装・レビュー・テスト修正”を全部やらせない**ことです。
小さく区切った方が、トークンも差分もレビュー負荷も減ります。

---

## 依頼時のガードレール

毎回入れておくとよい制約はこれです。

```text
- まず関連箇所を調査し、実装前に計画を提示すること
- 変更は最小差分にすること
- 共有ヘッダ変更は事前承認なしに行わないこと
- 構造体レイアウト変更は事前承認なしに行わないこと
- グローバル変数追加は事前承認なしに行わないこと
- 既存の初期化順序を変えないこと
- 関係ないリファクタリング、整形、命名変更をしないこと
- VC6で通るCとして書くこと
- 最後に変更ファイル、変更関数、リスク、テスト観点を報告すること
```

このあたりは、毎回コピペできる「AI利用ルール」としてリポジトリ直下に置いてもよいです。
たとえば `AI_AGENT_RULES.md` や `.github/copilot-instructions.md` 相当のファイルです。

## 小規模機能追加での理想的な依頼例

たとえば、こんな感じです。

```text
○○モジュールに、異常コード1234を検出した場合だけログを1行追加する小規模変更を行いたいです。

まずコード変更せず、既存の異常コード処理・ログ出力処理・類似ケースを調査してください。
対象はできれば○○モジュール内に閉じてください。

共有ヘッダ変更、構造体変更、グローバル変数追加が必要な場合は、実装せず報告してください。
既存の初期化順序、状態遷移、エラー処理は変えないでください。

出力は以下でお願いします。
1. 関連ファイル
2. 関連関数
3. 類似実装
4. 変更候補
5. 推奨案
6. リスク
7. テスト観点
```

その後に、

```text
推奨案1で実装してください。

共有ヘッダ、構造体、グローバル変数は変更しないでください。
関係ない整形はしないでください。
実装後に差分要約とテスト観点を出してください。
```

という二段階にします。

## 結論

新規の小規模機能追加では、AIエージェントには **「実装者」ではなく、最初は“調査員＋設計補助”として依頼する**のがよいです。

一番おすすめの型はこれです。

```text
調査だけ → 変更案 → 人間承認 → 最小実装 → 自己レビュー → 人間レビュー
```

この運用なら、27万トークンの大きな窓を毎回使い切らず、2〜6万トークン程度の局所コンテキストで多くの小規模変更を回せます。
特にVC6/C/大量グローバル/共有ヘッダありの保守開発では、**「変更範囲を狭くする依頼文」そのものが品質管理**になります。
